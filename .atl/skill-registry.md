### judgment-day
- Launch two independent blind judge sub-agents in parallel
- Each reviews the same target independently
- Synthesize findings, apply fixes, re-judge until both pass
- Escalate after 2 iterations if unresolved

### sdd-explore
- Investigate codebase and think through ideas
- Read relevant files to understand current state
- Compare approaches and document tradeoffs
- Return structured analysis without creating files

### sdd-propose
- Create change proposals with intent, scope, approach
- Include rollback plan for risky changes
- Identify affected modules/packages

### sdd-spec
- Write delta specifications with requirements and scenarios
- Use Given/When/Then format for scenarios
- Use RFC 2119 keywords (MUST, SHALL, SHOULD, MAY)

### sdd-design
- Create technical design with architecture decisions
- Include sequence diagrams for complex flows
- Document decisions with rationale

### sdd-tasks
- Group tasks by phase (infrastructure, implementation, testing)
- Use hierarchical numbering (1.1, 1.2, etc.)
- Keep tasks small enough to complete in one session

### sdd-apply
- Follow existing code patterns and conventions
- Use pnpm for all package operations
- Follow Clean Architecture: modules/domain/application/infrastructure
- Use TypeScript strict mode, Zod for validation

### sdd-verify
- Run tests if test infrastructure exists (Jest + Supertest for backend, RTL for frontend)
- Compare implementation against every spec scenario
- Report CRITICAL / WARNING / SUGGESTION levels

### sdd-archive
- Warn before merging destructive deltas (large removals)
- Sync delta specs to main specs
- Persist final state to engram
